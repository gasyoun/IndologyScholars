# План внедрения идей SciGuide в IndologyScholars

Дата: 2026-05-21  
Цель: использовать SciGuide как методологическую рамку для усиления качества данных, внешней связности и интерпретируемости архива `IndologyScholars`, не копируя тексты руководства и не подменяя локальный конференционный корпус внешними библиометрическими рейтингами.

## 1. Принципы внедрения

1. Не переносить материалы SciGuide дословно. В разделе "О руководстве" указано, что перепечатка без разрешения правообладателя запрещена; используем только идеи, структуру вопросов и ссылки на источники.
2. Разделять три уровня данных:
   - первичный корпус `IndologyScholars`: программы Зографских и Рериховских чтений;
   - кураторские нормализации: имена, организации, города, темы, медиа;
   - внешние справочники: ORCID, Wikidata, VIAF, ROR, OpenAlex, РИНЦ/eLIBRARY, Scopus, WoS ResearcherID, Google Scholar.
3. Не превращать архив конференций в рейтинг ученых. Допустимые показатели: участие, повторяемость, пересечения конференций, тематические профили, институциональные траектории. Нежелательные как основной слой: h-index, квартильность, цитатные ранги.
4. Любой внешний идентификатор хранить с источником, датой проверки и уровнем уверенности.
5. OpenAlex использовать осторожно: как открытый CC0-граф публикаций и кандидатных связей, но не как надежный источник годов рождения или полной биографии российских гуманитариев. В проекте уже зафиксирован отказ от OpenAlex birth-year proxy.
6. РИНЦ/eLIBRARY использовать как ручной или полуавтоматический reference workflow. Массовый скрейпинг не внедрять.

## 2. Что берем из SciGuide

| Блок SciGuide | Что полезно для проекта | Куда внедрять |
| --- | --- | --- |
| Данные и метаданные | Четкое различение первичных данных, метаданных и производных полей | `methodology.html`, `known-limitations.html`, README |
| Источники | Паспорт источников: охват, селективность, выгрузка, ограничения | `data-sources.html`, `datapackage.json` |
| Идентификаторы | ORCID, ROR, Wikidata, DOI/ISBN/ISSN, специализированные author IDs | `authority_ids.json`, JSON-LD, profile pages |
| API | OpenAlex/Crossref/ORCID как проверяемые внешние API | новые scratch/scripts для lookup queues |
| Тематическая классификация | Документировать, что тема присваивается на уровне доклада, а не человека | `themes/`, `data-quality.html`, article appendix |
| Сетевые методы | Сети соучастия, co-presence, человек-организация-тема | `analytics_output/`, future network page |
| Открытые данные | Лицензии, reuse, машиночитаемые выгрузки | `download-data.html`, `datapackage.json`, `CITATION.cff` |

## 3. Фаза 1: Документация и методологическая ясность

Приоритет: высокий.  
Смысл: сделать сайт понятным для исследователя, который хочет не только пользоваться поиском, но и понимать, что именно измеряется.

### Задачи

1. Расширить `methodology.html`.
   - Добавить блок "Данные, метаданные и производные поля".
   - Развести: программа конференции, извлеченный доклад, нормализованный участник, тематический код, аналитический показатель.
   - Отдельно указать, что тематика доклада не равна полной научной специализации человека.

2. Расширить `data-sources.html`.
   - Для каждого источника указать: тип источника, период охвата, статус кэша, способ обновления, известные риски.
   - Добавить раздел "Внешние справочники" с объяснением, что они не входят в первичный корпус.

3. Расширить `known-limitations.html`.
   - Добавить ограничения по:
     - идентификации персон по инициалам;
     - историческим аффилиациям;
     - тематическому кодированию;
     - неполноте OpenAlex для советской и российской гуманитаристики;
     - невозможности массового надежного использования РИНЦ без ручной проверки.

4. Добавить страницу или раздел "How to read the metrics".
   - Объяснить conference-native metrics: `total_talks`, `series overlap`, `newcomer rate`, `institution bridge`, `theme diversity`.
   - Прямо сказать, что это показатели участия в корпусе конференций, а не оценка качества научного вклада.

### Артефакты

- обновленные `methodology.html`, `data-sources.html`, `known-limitations.html`;
- при необходимости новый `metrics-guide.html`;
- ссылки на эти страницы из навигации и `sitemap.xml`.

### Критерии готовности

- исследователь может понять происхождение каждого важного поля без чтения Python-кода;
- на страницах нет обещаний точности, которые не подтверждаются текущими данными;
- ограничения сформулированы рядом с возможностями, а не спрятаны отдельно.

2. Расширить генераторы:

- все внешние ID проходят форматную проверку;
- страницы персон имеют `sameAs` только для подтвержденных ссылок;
- есть coverage-метрика: сколько персон и организаций имеют внешние ID.

## 5. Фаза 3: OpenAlex как кандидатный, а не авторитетный слой

Приоритет: средний.  
Смысл: использовать сильные стороны OpenAlex: открытый API, CC0, связи Works-Authors-Institutions-Concepts. Не использовать его там, где он уже показал риск.

### Что можно делать

1. Поиск кандидатных OpenAlex author profiles для персон.
2. Сохранение `openalex_author_id` только после ручной проверки.
3. Выгрузка кандидатных публикаций для подтвержденных OpenAlex IDs.
4. Сопоставление тем OpenAlex с локальными темами только как дополнительный сигнал.
5. Сбор DOI/источников публикаций для будущей библиографии профиля.

### Чего не делать

1. Не оценивать год рождения по первой публикации.
2. Не подменять локальную аффилиационную историю данными OpenAlex.
3. Не выводить citation counts как показатель качества участника конференций.
4. Не объединять людей автоматически только по OpenAlex search result.

### Артефакты

- `scratch/openalex_author_candidates.py`;
- `analytics_output/openalex_author_candidates.csv`;
- `analytics_output/openalex_confirmed_works.csv`;
- поле `openalex` в `authority_ids.json` для подтвержденных персон.

### Критерии готовности

- кандидатная таблица не меняет основной корпус автоматически;
- каждый принятый OpenAlex ID имеет ручной статус `confirmed`;
- OpenAlex explicitly documented as external enrichment, not primary source.

3. Добавить валидатор URL и ID для подтвержденных РИНЦ/SPIN записей.

### Артефакты

## 8. Фаза 6: Сетевые методы без лишней библиометрии

Приоритет: средний-низкий.  
Смысл: взять у SciGuide идею сетевого анализа, но строить сети на собственном корпусе.

### Возможные сети

1. Person-person:
   - совместные доклады;
   - участие в одной сессии;
   - участие в одной конференции в один год.
2. Person-organization:
   - аффилиационные траектории;
   - переходы между организациями;
   - bridge scholars между Зографскими и Рериховскими чтениями.
3. Person-theme:
   - устойчивость тем;
   - смена тематического профиля;
   - междисциплинарность.
4. Organization-theme:
   - какие организации чаще связаны с какими темами;
   - как это меняется по периодам.

### Задачи

1. Создать `analytics_output/network_edges.csv`.
2. Создать `analytics_output/network_nodes.csv`.
3. Добавить отдельные графики:
   - top bridge scholars;
   - institution-theme heatmap;
   - co-presence network by conference series.
4. Документировать, что это сети участия, а не сети цитирования.

### Артефакты

- `analytics_output/network_nodes.csv`;
- `analytics_output/network_edges.csv`;
- возможно `networks.html` или блок в dashboard.

### Критерии готовности

- каждое ребро сети имеет понятный тип и источник;
- веса ребер воспроизводимы из SQLite;
- сети не смешивают co-authorship, co-session и co-conference без явного `edge_type`.

## 11. Риски и меры контроля

| Риск | Что может пойти не так | Контроль |
| --- | --- | --- |
| Ошибочная идентификация персон | ORCID/OpenAlex/РИНЦ привязан к однофамильцу | `confidence`, `source`, ручная проверка, review queue |
| Переобещание библиометрической точности | Пользователь читает participation metric как оценку качества | `metrics-guide.html`, предупреждения на страницах |
| Нестабильные `presentation_id` | CSV с темами/видео теряют joinability после rebuild | детерминированные IDs или natural-key rematching |
| Неправомерный reuse SciGuide | Копирование текста руководства | только ссылки и собственные формулировки |
| РИНЦ scraping | блокировка/нарушение правил | только ручной workflow или официальная выгрузка/API |
| OpenAlex coverage gaps | ложные биографии и годы рождения | запрет birth-year proxy, external enrichment only |

## 14. Наукометрия конференций (Козицын, 2024)

Источник: Козицын А. С. О критериях ранжирования конференций // Конференция «Абрау-2024». ИПМ им. М. В. Келдыша РАН, 2024. URL: https://keldysh.ru/abrau/2024/temp/4.pdf

Обзор каталогов и рейтинговых систем научных конференций (CORE, Google Scholar h5, AMiner, GGS, ERA, CSCRS и др.) с формулировкой критериев ранжирования. Фокус — компьютерные науки (система ИСТИНА, МГУ), но набор критериев обобщаем.

### Что применимо к IndologyScholars

| Критерий Козицына | Наш аналог | Статус |
|---|---|---|
| Ретроспективные параметры (время существования, периодичность) | Серии 2004–2026 в `conferences.db` | ✅ |
| Географические параметры (места проведения) | `cities/*.html`, venues в БД | ✅ |
| Соотношение постоянного состава и новых участников | `newcomer_rate_by_year.csv`, `cohort_survival.csv`, `closedness_metrics.csv` | ✅ |
| Количество докладчиков | `total_talks` по событию | ✅ |

## 15. Кумулятивное преимущество и институциональный контекст (Ермолаева, 2026)

Источник: Ермолаева А. М. Механизмы кумулятивного преимущества в наукометрии // Диссертация на соискание ученой степени кандидата физико-математических наук. РУДН, 2026. URL: https://www.rudn.ru/storage/media/science_dissertation/85cfdd2e-6125-42d7-ab28-e973fd7d920b/LOBsxHu5QSzNrsJ6gxSgUmrOOt2ndf3j1zNxwTwc.pdf

Исследование Ермолаевой посвящено математическому моделированию кумулятивного преимущества (закона Матфея) в наукометрии.

### Применимость к IndologyScholars

1. **Закон Матфея (Matthew Effect) в структуре участия**:
   - Накопление докладов вокруг стабильного ядра (коэффициент Джини распределения докладов ≈ 0.46) представляет собой проявление кумулятивного преимущества. Стабильное ядро авторов привлекает внимание и новые контакты, формируя устойчивую структуру конференции. Использование структурных индексов (таких как индекс Джини и индекс закрытости) методологически обосновывается необходимостью измерения этой неравномерности.

2. **Обоснование статических метрик против цитатного лага**:
   - Ермолаева показывает, что классические цитатные метрики страдают от огромного временного лага накопления цитирований (среднее время полужизни цитирований cited half-life составляет 6–8 лет для естественных наук, а для гуманитарных дисциплин этот лаг существенно выше).
   - Это обосновывает отказ нашего проекта от динамических цитатных метрик в пользу статических метрик участия (общее число докладов, доля новичков, выживаемость когорт), которые доступны сразу по завершении мероприятия и с высокой точностью отражают структуру научного сообщества.

3. **Институциональные стимулы и КБПР (Комплексный балл публикационной результативности)**:
   - Внедрение методики КБПР в российских академических институтах с 2019 года привело к резкому снижению веса конференционных публикаций (тезисы и доклады оцениваются всего в 1 балл, тогда как статьи в ведущих журналах приносят от 2.5 до 20 баллов).
   - Этот дисбаланс дестимулирует регулярную подготовку и публикацию трудов (proceedings) конференций, заставляя организаторов издавать их выборочно или раз в несколько лет (как это происходит с Зографскими и Рериховскими чтениями). Данный контекст подчеркивает важность независимого архивирования и анализа метаданных докладов в рамках нашего проекта для фиксации реальной научной активности, остающейся невидимой для официальных баз.
